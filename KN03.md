### 6. Gemeinsamkeiten und Unterschiede

**Gemeinsamkeiten:**
- Container im gleichen Netzwerk können sich gegenseitig via IP erreichen
- Beide Netzwerke haben einen Default-Gateway (die jeweilige Bridge-IP)
- Container aus verschiedenen Netzwerken sind voneinander isoliert – weder via IP noch via Name erreichbar

**Unterschiede:**

| Merkmal | default bridge | user-defined (tbz) |
|---------|---------------|-------------------|
| DNS-Auflösung via Containername | ❌ Nein | ✅ Ja |
| Subnet konfigurierbar | ❌ Nein | ✅ Ja |
| Automatisches DNS | ❌ Nein | ✅ Ja |

**Schlussfolgerung:**  
Docker-Netzwerke sind voneinander isoliert. Container in verschiedenen Netzwerken können sich nicht erreichen – weder via IP noch via Name. Im default bridge Netzwerk gibt es kein automatisches DNS, weshalb `ping busybox2` fehlschlägt, obwohl beide Container im gleichen Netzwerk sind. Im user-defined Netzwerk (`tbz`) funktioniert die Namensauflösung automatisch, weil Docker dort einen eingebauten DNS-Server bereitstellt.

## B) Rückblick KN02

### In welchem Netzwerk befanden sich Web- und DB-Container?

Die Container befanden sich im **default bridge Netzwerk**, da kein eigenes Netzwerk angegeben wurde.

### Weshalb funktionierte die Verbindung über die IP-Adresse des DB-Containers?

Beide Container waren im gleichen Netzwerk. Innerhalb eines Docker-Netzwerks können Container sich gegenseitig direkt via IP erreichen – daher war die Verbindung zum DB-Container über seine IP möglich.

### Weshalb ist diese Lösung mit einer direkt eingetragenen Container-IP nicht ideal?

Docker vergibt IP-Adressen dynamisch beim Start. Nach einem `docker stop` / `docker start` oder Neustart kann ein Container eine andere IP erhalten. Die hartcodierte IP im Web-Container würde dann ins Leere zeigen und die Verbindung zur Datenbank schlägt fehl.

### Verbesserungsvorschlag

Ein **user-defined Netzwerk** erstellen und beide Container darin starten. Dann kann der Web-Container den DB-Container via **Containername** ansprechen. Docker löst den Namen automatisch zur aktuellen IP auf – unabhängig davon, ob sich die IP nach einem Neustart ändert.

## C) Verbesserung umsetzen

### Anpassung db.php

Der Wert von `$db_server` wurde von der hardcodierten IP `172.17.0.3` auf den Containernamen `kn02b-db` geändert:

```php
// Vorher
$db_server = "172.17.0.3";

// Nachher
$db_server = "kn02b-db";
```

### Befehle

```bash
# Alte Container stoppen und löschen
docker stop kn02b-web kn02b-db
docker rm kn02b-web kn02b-db

# Eigenes Netzwerk erstellen
docker network create kn02b-net

# DB-Image bauen und Container starten
cd C:\Users\dgdec\Desktop\kn02b-db
docker build -t kn02b-db-img .
docker run -d --name kn02b-db --network kn02b-net kn02b-db-img

# Web-Image bauen und Container starten
cd C:\Users\dgdec\Desktop\kn02b-web
docker build -t kn02b-web-img .
docker run -d --name kn02b-web --network kn02b-net -p 8083:80 kn02b-web-img
```
