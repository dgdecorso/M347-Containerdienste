## 9) Verbesserung 1 – Multistage Dockerfile & Runtime-Env

### Multistage Dockerfile

Der manuelle `npm run build` Schritt entfällt, da der Build vollständig im Image stattfindet:

```dockerfile
FROM node:20-alpine AS build
WORKDIR /app
COPY app/package*.json ./
RUN npm install
COPY app/ .
RUN npm run build

FROM nginx:alpine
WORKDIR /usr/share/nginx/html
COPY --from=build /app/build .
COPY env.sh /docker-entrypoint.d/40-env.sh
RUN chmod +x /docker-entrypoint.d/40-env.sh
EXPOSE 80
```

### Screenshot – Build-Output

<!-- Screenshot: docker buildx build Output (Multistage) -->

### Runtime Environment Variablen

Da React-Env-Variablen normalerweise beim Build hartcodiert werden, generiert ein Skript (`env.sh`) beim Container-Start eine `env.js`, die per `window._env_` zur Laufzeit gelesen wird. Dadurch lassen sich URLs über Kubernetes setzen, ohne das Frontend neu zu bauen.

## 10) Verbesserung 2 – LoadBalancer

Der Frontend-Service wurde von `NodePort` auf `LoadBalancer` umgestellt:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: frontend-service
spec:
  type: LoadBalancer
  selector:
    app: frontend
  ports:
    - port: 80
      targetPort: 80
```

```bash
microk8s kubectl apply -f frontend.yaml
microk8s kubectl get svc
```

### Screenshot – Service vorher (NodePort) vs. nachher (LoadBalancer)

<!-- Screenshot: frontend.yaml Diff -->

### Screenshot – kubectl get svc mit EXTERNAL-IP

<!-- Screenshot: get svc Output, EXTERNAL-IP evtl. <pending> bei MicroK8s -->

**Erklärung:** Ein LoadBalancer verteilt Anfragen automatisch auf verfügbare Nodes, statt dass Benutzer eine fixe Node-IP mit NodePort aufrufen müssen. Bei MicroK8s bleibt `EXTERNAL-IP` oft auf `<pending>`, da kein Cloud-Provider-Controller automatisch eine AWS-LoadBalancer-Instanz bereitstellt — in einer vollständigen Cloud-Kubernetes-Umgebung (z.B. EKS) würde dies automatisch geschehen.
