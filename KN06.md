## A) Installation – MicroK8s Cluster (3 Nodes)

### Installationsschritte

Auf allen drei EC2-Instanzen (Ubuntu 22.04, t2.medium):

```bash
sudo snap install microk8s --classic
sudo usermod -a -G microk8s ubuntu
sudo chown -f -R ubuntu ~/.kube
newgrp microk8s
microk8s status --wait-ready
```

Node 1 – Join-Token generieren:

```bash
microk8s add-node
```

Node 2 & Node 3 – Dem Cluster beitreten:

```bash
microk8s join <IP-Node1>:25000/<TOKEN>
```

### `microk8s status` – HA Cluster (3× Control Plane)

```bash
microk8s status
```

<!-- Screenshot: microk8s status mit high-availability: yes -->
<img width="1765" height="520" alt="image" src="https://github.com/user-attachments/assets/a207f9ee-fb43-4d37-9cbc-76404c521434" />

**Erklärung:**

Die ersten Zeilen von `microk8s status` zeigen, ob MicroK8s läuft und ob High Availability aktiv ist. Sobald mindestens drei Nodes dem Cluster beigetreten sind, aktiviert MicroK8s automatisch HA. Als Datenbank nutzt MicroK8s intern **dqlite** (distributed SQLite), das nach dem Raft-Konsensus-Prinzip funktioniert: alle drei Nodes verwalten den Cluster-State gemeinsam, sodass kein Single Point of Failure entsteht. Unter `datastore master nodes` sind deshalb alle drei Node-Adressen aufgelistet.

### `microk8s status` – 1 Master + 2 Worker

```bash
microk8s status
```

<!-- Screenshot: microk8s status mit high-availability: no -->
<img width="1368" height="252" alt="image" src="https://github.com/user-attachments/assets/0f5cd6ac-7414-4461-90ed-3544f1296ac9" />

**Erklärung:**

Im Vergleich zur vorherigen Ausgabe zeigt `microk8s status` jetzt `high-availability: no`. Unter `datastore master nodes` erscheint nur noch Node 1. Der Unterschied kommt daher, dass Worker-Nodes zwar Workloads ausführen, aber **nicht** am Cluster-State (dqlite) teilnehmen. Nur der Master-Node verwaltet die Kubernetes-Datenbank. Da weniger als drei Control-Plane-Nodes vorhanden sind, ist kein Raft-Quorum mehr möglich – HA wird deshalb automatisch deaktiviert.

## Unterschied `microk8s` vs. `microk8s kubectl`

`microk8s` ist das Verwaltungs-Tool für den MicroK8s-Daemon selbst. Damit wird der Node-Dienst gesteuert sowie die Cluster-Konfiguration verwaltet – zum Beispiel mit `microk8s start`, `microk8s status` oder `microk8s add-node`.

`microk8s kubectl` ist der eingebettete Kubernetes-Client, der mit dem API-Server des Clusters kommuniziert. Damit werden Kubernetes-Ressourcen innerhalb des Clusters verwaltet, wie Nodes, Pods oder Services. Kurz gesagt: `microk8s` betrifft den Node selbst, `microk8s kubectl` betrifft den Cluster-Inhalt.
